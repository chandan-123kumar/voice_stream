## ЁЯдЦ Claude

It's plain HTTP chunked transfer тАФ a single long-lived `POST /generate` request, not WebSockets, SSE, or WebRTC.

The flow:

- **Server side** (`app.py:143`): FastAPI returns a `StreamingResponse` wrapping the `pcm_stream_megakernel()` generator with `media_type="application/octet-stream"`. Each chunk the generator yields is raw PCM16 mono 24kHz audio. Since no `Content-Length` is known up front, uvicorn sends it with HTTP/1.1 `Transfer-Encoding: chunked`.
- **Browser side** (`app.py:246-268`): the page uses the Fetch API's streaming body тАФ `resp.body.getReader()` on a `ReadableStream` тАФ to consume chunks as they arrive, converts each Int16 chunk to Float32, and schedules it for playback through the Web Audio API (`AudioContext.createBufferSource`).

So it's one-directional serverтЖТbrowser streaming over an ordinary HTTP response, which is why TTFA can be low: the browser starts playing as soon as the first chunk lands rather than waiting for the full file. Also note the server binds `127.0.0.1:8000`, so the whole stream actually rides through your SSH tunnel (`ssh -L 8000:localhost:8000`) from the 5090 box to your machine.

## ЁЯдЦ Claude

For this assignment there are really two hops, and they want different answers:

## Megakernel server тЖТ Pipecat: in-process, or WebSocket if separate

The best "protocol" is **no protocol at all**: implement a Pipecat `TTSService` subclass that wraps your `MegakernelTTS` engine directly in the same process. Pipecat services are just Python classes тАФ in `run_tts()` you iterate `engine.stream(...)` (in a thread executor, since Pipecat is asyncio and your generation is blocking) and yield `TTSAudioRawFrame`s chunk-by-chunk. Everything runs on the one 5090 anyway, so a network hop between the server and Pipecat buys you nothing and costs you latency, serialization, and a process boundary. This is the easiest way to hit TTFC < 50 ms and satisfies "streaming frame-by-frame, not buffered-then-sent" trivially.

If you do want the standalone inference server (Step 2 asks for one тАФ you can deliver it *and* still integrate in-process), make it **WebSocket**, not your current HTTP chunked endpoint. Two reasons that matter specifically for voice agents:

1. **Interruption (barge-in).** When the user starts talking over the bot, Pipecat cancels TTS mid-utterance. Over WebSocket you send a cancel message on the same connection; with HTTP chunked your only option is tearing down the connection, and the next utterance pays connection setup again.
2. **Persistent connection.** A voice turn is many short TTS requests. One long-lived WebSocket means each utterance's first byte isn't behind a fresh request handshake тАФ that's TTFC budget. This is also exactly how the production streaming TTS services Pipecat already integrates (Cartesia, ElevenLabs, Deepgram) do it, so you have reference `TTSService` implementations to crib the shape from.

HTTP chunked (what `app.py` does now) is fine to keep for the demo page, but it's the wrong fit for the Pipecat leg.

## Pipecat тЖТ browser/mic: WebRTC

For the user-facing leg of Step 4 (speak тЖТ ... тЖТ audio playback), use Pipecat's **`SmallWebRTCTransport`** (or Daily). WebRTC is purpose-built for real-time bidirectional audio: Opus encoding, jitter buffering, echo cancellation for your mic input, and graceful handling of timing variation тАФ all things raw WebSocket/HTTP audio makes you handle yourself, and which directly affect the "no glitches, dropped frames" criterion. Pipecat ships a prebuilt web client for it, so it's also the least work.

So the stack for the deliverable: **mic/browser тЗД WebRTC тЗД Pipecat pipeline (STT тЖТ LLM тЖТ your in-process megakernel TTSService) тЗД WebRTC тЗД speaker**, with an optional standalone WebSocket TTS server as the documented "streaming interface" artifact.

## ЁЯдЦ Claude

Done and verified. The in-process service is implemented in `qwen_tts_megakernel/pipecat_tts.py` and passes an end-to-end pipeline smoke test.

**What was built**

- `MegakernelTTSService(TTSService)` (qwen_tts_megakernel/pipecat_tts.py) тАФ modeled on Pipecat's local Piper service, the closest first-party analogue:
  - `run_tts()` runs the blocking `engine.stream()` loop in a worker thread holding a `threading.Lock` (single KV cache), and hands PCM16 chunks to the asyncio side through a queue the moment the codec decodes them. Frames go downstream via Pipecat's `_stream_audio_frames_from_iterator`, which handles resampling and audio contexts.
  - Interruption-safe: when Pipecat cancels the generator mid-utterance (barge-in), the worker sees the cancel flag at the next chunk boundary, stops decoding, and releases the engine lock.
  - Speaker maps to Pipecat's `voice` setting, and `language_to_service_language()` maps Pipecat `Language` enums (any regional variant, e.g. `en-US`) to Qwen's language names. Both are runtime-updatable via `TTSUpdateSettingsFrame`.
  - `gen_params` passes through engine knobs (`temperature`, `first_chunk_frames`, `chunk_frames`, тАж), and you can hand it a pre-warmed engine plus a shared lock if `app.py` runs in the same process.
- `tests/test_pipecat_tts.py` тАФ minimal `Pipeline([tts, collector])` that pushes a `TTSSpeakFrame` and asserts incremental streaming.

**Test results** (engine warm, alongside the running app.py):

- 12 incrementally streamed `TTSAudioRawFrame`s тАФ frame-by-frame, not buffered-then-sent
- Pipecat-measured TTFB **164 ms**; 10.9 s of audio generated in 1.95 s тЖТ **RTF тЙИ 0.18**
- Output written to `/tmp/pipecat_tts_test.wav` for a listen

I also installed `pipecat-ai` 1.3.0 plus `websockets` (worth adding to `requirements.txt`).

One note on the TTFC < 90 ms target: the 164 ms includes the HF prefill forward plus the default `first_chunk_frames=4` before the first codec decode. Passing `gen_params={"first_chunk_frames": 2}` (or 1) is the first knob to try when you get to the benchmarking step тАФ smaller first chunk trades a bit of decode efficiency for faster first audio.

## ЁЯдЦ Claude

TTFB is now under the 90 ms target. The chunk-size knob alone wasn't enough, though тАФ the real win came from fixing a thread-reuse problem it exposed.

**What I found**: with `first_chunk_frames=2` the TTFB only dropped from ~124 ms to ~115 ms, and `=1` barely helped. Instrumenting the service showed the whole delay was inside the worker, with even the prefill running ~10 ms slower than in my raw-engine benchmark (where warm TTFA was ~62 ms). An isolation test pinned it down: a **fresh thread per utterance costs ~35 ms** because PyTorch keeps CUDA state (cuBLAS handles, etc.) per-thread, so every new thread re-initializes it on its first kernels. `asyncio.to_thread` with its reused pool thread matched main-thread speed.

**The fix** (in `qwen_tts_megakernel/pipecat_tts.py`): the service now owns a persistent single-thread `ThreadPoolExecutor` for all generations instead of spawning a `threading.Thread` per utterance, and runs a short warmup generation on that thread at init (`warmup=True` by default) so even the first real utterance is fast. It's shut down in `cleanup()`.

**Measured TTFB through the Pipecat pipeline** (two utterances each):

| first_chunk_frames | TTFB |
|---|---|
| 4 (engine default) | 102 ms |
| 2 | 82тАУ87 ms тЬУ |
| 1 | 72тАУ75 ms тЬУ |

I set `first_chunk_frames=2` as the service default тАФ comfortably under 90 ms while keeping the first codec decode a bit less aggressive than a single frame; passing `gen_params={"first_chunk_frames": 1}` buys another ~10 ms if you want it. Worth a listen to `/tmp/pipecat_tts_test.wav` to confirm the smaller first chunk doesn't introduce an audible seam at utterance start.

**Why WebSocket is fine here:**

- Pipecat ships `FastAPIWebsocketTransport` / `WebsocketServerTransport` as first-class transports, with matching JS client SDK support тАФ so the pipeline wiring (`transport.input() тЖТ STT тЖТ LLM тЖТ TTS тЖТ transport.output()`) is identical, just a different transport class. It even slots into your existing FastAPI `app.py`.
- A WebSocket is just TCP, so it rides your `ssh -L 8000:localhost:8000` tunnel with zero extra setup. **WebRTC mostly doesn't** тАФ its media goes over UDP/ICE, which an SSH local-forward can't carry. To use WebRTC with a rented Vast.ai box you'd need direct UDP reachability to the server or a TURN relay тАФ real friction for a one-day assignment.
- The things WebRTC is genuinely better at тАФ jitter buffering, packet-loss concealment, Opus compression, adapting to bad networks тАФ barely matter on a clean localhost-tunneled link with one user. Raw PCM at 24 kHz mono is only ~384 kbps, trivial for the tunnel.
- The voice-agent essentials are transport-agnostic anyway: VAD and interruption run server-side in Pipecat (Silero), and browser echo cancellation happens at `getUserMedia`, not in the transport.

**When you'd still want WebRTC:** a production deployment where real users connect over the open internet тАФ there TCP head-of-line blocking turns packet loss into audio stutters and growing latency, and that's exactly what WebRTC's UDP stack fixes. Worth one honest line in your README ("WebSocket transport chosen because the demo runs over an SSH tunnel; WebRTC is the production path"), which also scores on the "communication/honest about tradeoffs" evaluation criterion.

So: build the demo on `FastAPIWebsocketTransport`, mention WebRTC as the production upgrade path, and move on.
